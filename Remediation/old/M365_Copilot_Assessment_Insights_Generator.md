# Microsoft 365 Copilot Assessment — Insights Generator

---

## About This File

### Purpose

This file is a **prompt engineering guide** that drives the automated generation of enriched deliverables from M365 Copilot readiness assessment data. It transforms raw assessment Excel output into stakeholder-ready materials: executive summaries, detailed action plans, project trackers, email templates, and priority-segmented documents.

### Input

Any Excel file (`.xlsx`) produced by this workspace's assessment tool, containing the following columns:

| Column | Type | Required | Description |
|--------|------|----------|-------------|
| `Service` | String | Yes | Service category (e.g., M365, Entra, Defender) |
| `Feature` | String | Yes | Assessed feature name |
| `Status` | String | Yes | Assessment result (Success, Action Required, Insight, Warning, PendingActivation, Access Denied) |
| `Priority` | String | Conditional | Critical/High/Medium/Low — present only for non-Success rows |
| `Observation` | String | Yes | Assessment finding text |
| `Recommendation` | String | Conditional | Remediation guidance — present only for non-Success rows |
| `Link Text` | String | Optional | Reference documentation label |
| `Link URL` | String | Optional | Reference documentation URL |

> **Note:** The input file is generated by the main assessment tool (`main.py`) and saved to `Reports/`. Any Excel following this schema will work regardless of tenant, date, or assessment scope.

### Output

15 files generated into `Remediation/Output/`:

| # | File | Format | Description |
|---|------|--------|-------------|
| 1 | `Executive_Summary.docx` | Word | Stakeholder-ready executive summary (~6 pages) |
| 2 | `Copilot_Readiness_Action_Plan.md` | Markdown | Detailed per-item remediation guide (~70-100 KB) |
| 3 | `Copilot_Readiness_Tracker.xlsx` | Excel | 5-sheet project tracker with formatting and validation |
| 4 | `Email_Templates.txt` | Text | 2 audience-specific delivery emails (IT Manager + Executive) |
| 5 | `High_Priority_Action_Plan.docx` | Word | Self-contained document for High priority items |
| 6 | `Medium_Priority_Action_Plan.docx` | Word | Self-contained document for Medium priority items |
| 7 | `Low_Priority_Action_Plan.docx` | Word | Self-contained document for Low priority items |
| 8-11 | `*.png` | PNG | 4 professional charts (status, priority, services, actions) |
| 12-15 | `*.json` | JSON | Machine-readable data backing each chart |

## Risk Assessment Heuristic Logic

Columns 8-11 in the tracker (Telemetry Limitation, False Positive Risk, False Negative Risk, Confidence in Observation) are **generated by the script** — they do not exist in the source Excel. The `assess_risk(feature, observation, status)` function applies contextual heuristic reasoning based on three inputs:

| Input | Source | Example |
|-------|--------|---------|
| `feature` | Feature column | "Conditional Access Policies" |
| `observation` | Observation column | "0 of 5 users enrolled in MFA" |
| `status` | Status column | "Action Required" |

### Decision Matrix

| Column | Logic | Output Format |
|--------|-------|---------------|
| **Telemetry Limitation** | Detects access-denied states (Yes — permissions missing), zero-count anomalies (Possible — zero metric may reflect data lag), configuration-based findings (No — confirmed directly via admin API), and partial data latency | `"Yes/No/Possible — [contextual explanation]"` |
| **False Positive Risk** | Evaluates whether the observation could be misleading: binary config states → Low; quantitative metrics with specific counts → Low; zero-count values → Medium (could be data lag); permission gaps → High (cannot verify) | `"Low/Medium/High — [reasoning]"` |
| **False Negative Risk** | Considers feature-specific blind spots: CA custom names or third-party IdP → Medium; BYOD/unmanaged endpoints → Medium; co-managed Intune/SCCM → Medium; third-party DLP overlap → Medium; deterministic absence → Low | `"Low/Medium/High — [reasoning]"` |
| **Confidence** | Rates based on metric type: concrete percentages → High; specific non-zero counts → High; zero-counts with clear observation → Medium-High; partial/inferred states → Medium; access-denied → Low | `"High/Medium-High/Medium/Low — [reasoning]"` |

### Feature-Specific Heuristic Rules

| Feature Keywords | Telemetry | FP Risk | FN Risk | Confidence |
|-----------------|-----------|---------|---------|------------|
| `conditional access`, `ca polic` | No — config state is deterministic | Low — absence of policy is binary | Medium — custom-named or third-party IdP policies may exist | High — policy enumeration is complete |
| `mfa`, `multi-factor` | No — enrollment data from authentication methods API | Low — registration state is factual | Medium — users may have external MFA (e.g., Duo) | High if % present; Medium if zero-count |
| `sharepoint`, `onedrive`, `content` | Possible — usage reports have 48h latency | Medium — site count may exclude personal sites | Medium — content may exist in third-party platforms | Medium-High if specific count; Medium if zero |
| `defender`, `endpoint`, `xdr`, `device` | Possible — device telemetry depends on agent health | Low — onboarding count is factual | Medium — BYOD/unmanaged endpoints not visible | High if device count present; Medium otherwise |
| `sensitivity label`, `dlp`, `information protection` | No — label policy config is deterministic | Low — policy absence is binary | Medium — third-party DLP may provide coverage | High — policy existence confirmed via API |
| `purview`, `compliance`, `audit` | Possible — audit log ingestion may lag | Medium — partial data could mask scope | Medium — multi-tenant or hybrid gaps | Medium |
| `teams`, `meeting`, `collaboration` | Possible — usage reports have 48h latency | Medium if zero-count — could be data lag | Low — Teams is primary collaboration surface | High if usage metrics present; Medium if zero |
| `intune`, `device management`, `compliance` | Possible — sync cadence depends on device check-in | Low — enrollment count is factual | Medium — co-managed SCCM devices may not report | High if count > 0; Medium if zero |
| `access denied` (status) | Yes — cannot verify due to missing permissions | High — data unavailable; finding may be invalid | High — true state unknown | Low — no data to base conclusion on |
| *(generic fallback)* | Possible — depends on API data freshness | Medium — standard assessment heuristic | Medium — standard assessment heuristic | Medium — based on available signals |

## Step 2: Detailed Row-by-Row Implementation Guide

**Output file**: `Copilot_Readiness_Action_Plan.md` (~80-100 KB)  
**Purpose**: Complete remediation guidance for every non-success item, detailed enough for junior administrators to execute

### Prompt

> Generate a detailed, step-by-step remediation guide for all entries in the uploaded file that do not have "success" status.

### Source Data Requirements

- Excel file with columns: Service, Feature, Status, Priority, Observation, Recommendation, Link Text, Link URL, Risk Assessment fields
- Filter: Include only rows where Status ≠ "Success"

### Document Structure

#### Cover Section

- **Title**: "Microsoft 365 Copilot Readiness - Detailed Action Plan"
- Generation date
- Total items count
- Priority breakdown (Critical/High/Medium/Low with counts)
- Table of contents with anchor links, organized by priority

#### Per-Item Structure (sorted by Priority: Critical → High → Medium → Low)

For **each** non-success item, generate ALL of the following sections:

##### A. Item Header
- Item number (sequential)
- Service category badge
- Feature name
- Priority indicator (color-coded in final output)
- Status indicator

##### B. Current Situation
- Full observation text from source data
- Context paragraph: why this matters for Copilot readiness

##### C. Recommended Action
- Full recommendation text from source data
- Detailed explanation of approach

##### D. Risk Assessment Table

| Field | Description |
|-------|-------------|
| Telemetry Limitation | Whether data collection may be incomplete |
| False Positive Risk | Risk that the finding is incorrect |
| False Negative Risk | Risk that additional issues exist undetected |
| Confidence Level | Confidence in the accuracy of the observation |
| Impact if not addressed | Business consequence of inaction |

> **Note**: These columns are **not present in the source Excel**. The script's `assess_risk(feature, observation, status)` function generates them using heuristic reasoning. See [Risk Assessment Heuristic Logic](#risk-assessment-heuristic-logic) below for the full decision matrix.

##### E. Step-by-Step Implementation (Context-Aware)

Generate steps based on **feature type** — do NOT use generic steps when a specific category applies:

**For Conditional Access policies:**
1. Portal navigation (exact URL: https://entra.microsoft.com → Protection → Conditional Access)
2. Policy creation steps with specific settings
3. Configuration details (users, apps, conditions, grant controls)
4. Testing approach (Report-only mode, 7+ day duration)
5. Validation steps (sign-in logs review)
6. Enablement process (switch from Report-only to On)
7. Documentation requirements

**For MFA enrollment:**
1. Portal access steps (Entra → Security → Authentication methods)
2. MFA method enablement (Microsoft Authenticator, FIDO2, etc.)
3. User enrollment process and registration campaign
4. Conditional Access integration for enforcement
5. Communication plan (email templates, help articles)
6. Help desk preparation (troubleshooting guide)
7. Compliance monitoring (registration report)

**For SharePoint/content deployment:**
1. Current state assessment (sites, document libraries)
2. Migration planning (content inventory, priority)
3. Site creation steps (Teams-connected vs standalone)
4. Content migration tools (SharePoint Migration Tool, Mover)
5. Governance configuration (sensitivity labels, DLP)
6. Training requirements (end-user adoption)
7. Adoption metrics (usage reports, Copilot search validation)

**For Security/Defender services:**
1. Service activation (Defender portal → Settings)
2. Device onboarding methods (Intune, GPO, script)
3. Policy configuration (EDR, attack surface reduction)
4. Monitoring setup (alerts, incidents)
5. Alert configuration (email notifications, severity)
6. Team training (SecOps procedures)

**For OAuth/application security:**
1. Current app inventory (Entra → Enterprise apps)
2. Risk assessment criteria (permissions scope, publisher)
3. Review process (consent review workflow)
4. Revocation steps (remove high-risk app consents)
5. Policy configuration (admin consent workflow)
6. Ongoing monitoring (app governance)

**For Access Reviews:**
1. Create access review (Entra → Identity Governance)
2. Configure scope (groups, apps, roles)
3. Set recurrence (quarterly recommended)
4. Assign reviewers (managers, resource owners)
5. Configure auto-actions (remove if denied/no response)
6. First cycle validation
7. Reporting and compliance evidence

**For Passwordless authentication:**
1. Enable authentication methods (FIDO2, Windows Hello, Authenticator passwordless)
2. Pilot group selection
3. Registration campaign configuration
4. User onboarding process
5. Conditional Access strength policies
6. Feedback collection
7. Metrics tracking (passwordless sign-in %)

**For Cross-tenant access:**
1. Review current cross-tenant settings (Entra → External Identities)
2. Define trust policies (inbound/outbound)
3. Configure B2B collaboration settings
4. MFA trust configuration
5. Conditional Access for external users
6. Testing with partner organizations
7. Monitoring and logging

**For Intune/device management:**
1. Intune enrollment configuration
2. Compliance policy creation
3. Device configuration profiles
4. App protection policies
5. Conditional Access device compliance requirement
6. Reporting and monitoring
7. User communication

**For Group-based licensing:**
1. Create/identify target groups in Entra ID
2. Assign licenses to groups
3. Validate license assignment (processing status)
4. Handle assignment errors (conflicts, insufficient licenses)
5. Remove direct user assignments
6. Monitor license utilization
7. Document group-license mapping

**For Internet/Private Access (Global Secure Access):**
1. Enable Global Secure Access in Entra
2. Configure traffic forwarding profiles
3. Deploy Global Secure Access client
4. Configure security profiles
5. Test connectivity and access
6. Monitor traffic logs
7. Refine policies based on usage

**For Kaizala/frontline migration:**
1. Assess current Kaizala usage
2. Plan migration to Teams
3. Configure Teams for frontline workers
4. Migrate communication groups
5. Train frontline managers
6. Decommission Kaizala
7. Validate Teams adoption metrics

**Generic fallback (when no specific category matches):**
1. Review current state in relevant admin portal
2. Consult official Microsoft documentation
3. Plan implementation approach
4. Configure in test/pilot scope
5. Validate changes work as expected
6. Roll out to production
7. Monitor for issues
8. Document completion

## Appendix: Missing Data & Assumptions

### When Source Data Lacks Risk Assessment Columns

If the Excel file does not contain `Telemetry Limitation`, `False Positive Risk`, `False Negative Risk`, or `Confidence` columns:
- Populate these fields with: **"Not assessed — manual review recommended"**
- Include a note in the Data Quality Assessment section
- Flag in the Executive Summary under "Limitations"

### Environment Assumptions

| Assumption | Impact | Recommendation |
|-----------|--------|----------------|
| Specific M365 license SKUs not captured | Generic guidance | Capture SKUs in future assessments |
| Organization size unknown | Generic timelines | Adjust timelines to actual capacity |
| Risk tolerance not stated | Standard risk thresholds applied | Confirm with customer |
| Change freeze periods not captured | Timelines may conflict | Ask about blackout periods |

### Data Quality Indicators

- "0 of 0 users" metrics may indicate incomplete telemetry rather than actual zero state
- Access Denied (403) responses indicate permission gaps in data collection
- PendingActivation items may resolve without intervention
